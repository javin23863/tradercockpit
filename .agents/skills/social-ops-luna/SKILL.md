# Luna social operations

Delegate repetition, not discovery. Sol owns the first successful run, workflow changes, acceptance
criteria, and final review. Luna owns later executions inside the proven contract.

## Quality ownership

Sol remains ultimately responsible for consumer-facing quality and pipeline implementation after
delegation. Treat every Luna output as a candidate until Sol reviews the actual script, evidence,
charts, cuts, audio, final export, packaging, and receipts against the unchanged gates. Luna cannot
self-certify quality or lower a threshold. A quality regression returns to Sol for root-cause pipeline
repair before another delegated run. Sol acceptance does not replace the operator's exact-hash
publication approval; both are required.

## Delegation gate

Delegate only when all are true:

- The saved Codex project resolves exactly to this repository and canonical folder.
- The current branch/worktree, inputs, commands, outputs, and acceptance checks are known.
- One real end-to-end run has passed; fixtures or dry runs alone do not establish the lane.
- Authentication already exists and the task needs no password, MFA, provider terms, or new account.
- Public writes are either forbidden or bound to a current exact-hash approval.
- The exact script hash has a separate operator approval receipt before TTS, asset capture,
  assembly, or render; claims/style/editorial tool passes cannot replace that gate.

Keep a first-time provider setup, credential change, workflow redesign, unsupported claim, or failed
acceptance gate with Sol. Do not ask Luna to infer missing authority.

## Dispatch

Use a project-scoped Codex task or automation with:

- Model `gpt-5.6-luna` and reasoning effort `xhigh` (Luna Extra High/Fast).
- The exact TraderCockpit project, never a projectless target.
- One lane, one branch/worktree, and one explicit authority boundary.
- A packet containing current branch/head/dirty state, exact commands, input and output paths,
  approvals/hashes, expected receipts, verification commands, secrets boundaries, stop conditions,
  and the exact candidate artifacts Sol must inspect.

Update an existing automation instead of creating a duplicate.

## Operating week

- Monday–Friday market authority: use `daily-news-video`, `market-analysis`, `godseye-footage`, and
  `tradercockpit-free-media`; execute only this repository's `OpenMontage/` checkout.
- Saturday weekly recap: use `weekly-market-recap`. Keep the first real run with Sol; move the
  recurring automation to Luna Extra High/Fast only after its end-to-end acceptance receipt exists.
- Sunday analytics: run `tools/social_analytics.py collect`, `tools/dashboard.py --no-open`, the
  focused tests, credential-key scan, process review, and vault refresh. Do not produce or publish.
- Email consent health: delegate only after Buttondown subscribe → confirm → unsubscribe passes with
  a real receipt. Recurring work may verify list/form health and draft copy; sending still requires
  current exact approval and an authenticated established path.

## Return to Sol

Stop and return the evidence when a run needs a new dependency, schema/workflow change, credential or
terms action, unsupported factual claim, different provider, revised public asset, or broader authority.
Luna must preserve existing uploads, avoid cross-repository edits, update the vault from live facts,
and report exact receipts rather than declaring success from generated manifests alone. Sol decides
whether the run passed and owns every corrective pipeline change.
